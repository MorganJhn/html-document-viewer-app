# Revision And Troubleshooting Playbook

Use this when the user asks for changes after reviewing in the app.

## If The User Provides `HDV_REF`

1. Parse the fields.
2. Open the referenced file under the active workspace.
3. Locate the component by `data-hdv-id`, `data-component`, `data-name`, `id`, or `path`.
4. Confirm tag and nearby text.
5. Make the requested source edit.
6. Preserve or improve metadata.
7. Tell the user what changed and where.

Example user request:

```text
Make this less prominent:
HDV_REF file="brief.html" path="0.1.4.2" tag="aside" id="risk-callout" component="Risk callout"
```

Expected agent behavior:

```text
Open workspace/brief.html, find data-hdv-id="risk-callout", adjust only that callout's relevant styles/content, preserve the ID, and ask the user to refresh the app.
```

## If The User Gives Only A Human Label

Search source files for:

```text
data-component="label"
data-name="label"
aria-label="label"
id="label"
```

If multiple matches exist, use document context or ask for the copied `HDV_REF`. Do not guess across repeated components.

## If A Reference Does Not Resolve

Possible causes:

- The document was edited after the reference was copied.
- The user selected an element generated by script.
- The active workspace changed.
- The app preview came from an exported file, not the source file.
- Component metadata was removed during a previous edit.

Resolution:

1. Search the referenced file for `id`, `component`, `name`, and visible text.
2. Search all workspace HTML files if the `file` field is wrong.
3. Compare against recent backups under `_hdv/backups`.
4. Ask the user to reselect and copy a fresh reference only after reasonable search fails.

## If Pagination Changes Unexpectedly

Check:

- `@page` rules,
- `<style data-hdv-document-settings>`,
- `break-after`, `break-before`, and `break-inside`,
- image dimensions,
- font loading,
- margins and page size,
- absolute or fixed positioning,
- content that grew after text edits.

Use the app preview and PDF export for final pagination verification when layout matters.

## If The User Made Live Edits In The App

The app writes live edits back to source HTML and creates a backup. Read the current source file before making additional edits. Do not assume your earlier in-memory version is still current.

Backups are under:

```text
workspace/_hdv/backups/
```

## If A Component Should Be Reused

Prefer this flow:

1. Ask the user to select the component and save it to the Component library, or do it through the app if you are operating the UI.
2. Insert it into target documents through the app.
3. If manual insertion is needed, copy the library record's `html` into the target document.
4. Update duplicate `data-hdv-id` and `id` values.
5. Verify the inserted component is selectable and named correctly.

## If The App Does Not See Documents

Check:

- active workspace path in the app,
- `.hdv/config.json`,
- file extension is `.html` or `.htm`,
- documents are not under `_hdv`,
- server is running at `http://127.0.0.1:4174`.

If the workspace was changed in the app, use that workspace rather than the repository default.

## If Remote Saves Fail

Remote saves require `HDV_TOKEN` on the server and the same token in the app. Localhost does not require a token.

For private-network use:

```bash
set HOST=0.0.0.0
set HDV_TOKEN=your-token
npm run dev
```

Then tell the user the reachable URL and token.

## If PDF Export Fails

Check:

- Playwright/Chromium installation,
- server logs,
- asset paths,
- remote token configuration,
- CSS print rules,
- page size settings,
- whether the document references local absolute files that Chromium cannot load.

Run `npm run test:e2e` after app export changes.

## Verification For App Code Changes

Run:

```bash
npm test
npm run lint
npm run build
npm run test:e2e
```

For visual UI changes, also open the app and check a desktop viewport around `1366x900` or `1440x960`. Confirm the document browser, paged canvas, selector popover, inspector tabs, and export controls are visible and usable.
