# Component References

The copied component reference is the main communication bridge between user and agent. Human labels are useful for scanning, but the `HDV_REF` string is the authoritative pointer.

## Format

Example:

```text
HDV_REF file="sample-document.html" path="0.1.0.0" tag="h1" id="hero-title" component="Main headline" selector="data-hdv-id=hero-title"
```

Each selected component produces one line. Multi-selection copies multiple lines in selection order.

## Field Meanings

- `file`: workspace-relative source HTML file. Open this first.
- `path`: current parser path to the element in the HTML tree. Use as fallback or confirmation.
- `tag`: rendered element tag name.
- `id`: stable identifier from `data-hdv-id` when present, otherwise `id`.
- `component`: `data-component` value.
- `name`: `data-name` value.
- `selector`: best source selector hint generated by the app, such as `data-hdv-id=hero-title`.

Fields are omitted when the app cannot infer a value.

## Resolution Order

When you receive an `HDV_REF`, resolve it in this order:

1. Open `<active-workspace>/<file>`.
2. If `selector` starts with `data-hdv-id=`, search that exact `data-hdv-id`.
3. If `id` is present, search for `data-hdv-id="<id>"`, then `id="<id>"`.
4. Search exact `data-component`.
5. Search exact `data-name`.
6. Use `path` as a parser-tree fallback.
7. Confirm the `tag`, nearby text, page context, and parent component before editing.

Do not rely only on visible text when metadata is available. Many generated documents contain repeated labels like "Summary", "Total", or "Notes".

## Path Meaning

`path` is a dot-separated index path through element children in the parsed HTML document. It is precise for the current source, but it can shift after structural edits.

Example:

```text
path="0.1.0.2"
```

This means:

- first root element,
- second child element,
- first child element,
- third child element.

Use the app's `server/html.ts` runtime annotation logic as the implementation reference. The path is for locating a current tree node, not for naming a stable component.

## Source Selector Hints

The app annotates rendered elements with `data-hdv-source-selector` during preview. This attribute is runtime-only and should not be added manually to source documents.

Source documents should contain stable source metadata instead:

```html
<article data-component="Evidence card - fiscal policy" data-hdv-id="evidence-card-fiscal-policy">
  ...
</article>
```

Recommended `data-hdv-id` style:

- lowercase,
- hyphen-separated,
- unique within a document,
- descriptive enough to search,
- stable across copy edits.

Good:

```html
data-hdv-id="methods-summary"
data-hdv-id="chart-renewable-capacity-2025"
data-hdv-id="appendix-limitations-table"
```

Weak:

```html
data-hdv-id="box1"
data-hdv-id="new-section"
data-hdv-id="thing"
```

## Multi-Selection Handling

When the user selects multiple components, the copied text contains one `HDV_REF` per line:

```text
HDV_REF file="brief.html" path="0.1.2" tag="section" id="findings" component="Findings"
HDV_REF file="brief.html" path="0.1.3" tag="section" id="recommendations" component="Recommendations"
```

Handle each line independently. Preserve order when applying ordered requests such as "swap these", "make the first match the second", or "delete both selected blocks".

## Ambiguity Rules

If two elements have the same `data-component`, use `id`, `path`, parent context, and nearby text to disambiguate. If the reference still cannot be resolved, ask the user to select and copy a parent component or a nearby unique component.

If a referenced element was created only by runtime JavaScript and does not exist in source HTML, explain that the rendered state is not directly source-persistable and edit the script/template that produces it when possible.
